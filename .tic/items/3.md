## Summary

Add a Jira backend to tic using the [Atlassian CLI (acli)](https://developer.atlassian.com/cloud/acli/guides/introduction/) as the interface layer, following the same pattern as the existing GitHub (`gh`), GitLab (`glab`), and Azure DevOps (`az`) backends.

## Motivation

Jira is one of the most widely used issue trackers. Adding Jira support would make tic useful for a large number of teams that use Jira Cloud for project management.

## Configuration

Unlike other backends that auto-detect from git remotes, Jira projects aren't inherently tied to git repos. Configuration is explicit in `.tic/config.yml`:

```yaml
backend: jira
jira:
  site: https://mycompany.atlassian.net
  project: TEAM
  boardId: 6    # required for sprint/iteration support
```

- `site` — Full Jira Cloud URL
- `project` — Jira project key
- `boardId` — Scrum/Kanban board ID (required for sprint support)

Authentication is handled by `acli jira auth login` (OAuth via `--web` or API token via `--email`/`--token`).

## Capabilities

```typescript
getCapabilities(): BackendCapabilities {
  return {
    relationships: true,      // parent via --parent, deps via workitem link
    customTypes: false,        // types come from Jira project config
    customStatuses: false,     // statuses come from Jira workflow
    iterations: true,          // sprints (when boardId configured)
    comments: true,            // workitem comment create/list
    fields: {
      priority: true,
      assignee: true,
      labels: true,
      parent: true,
      dependsOn: true,
    },
  };
}
```

## ACLI Command Mapping

| Backend Method | ACLI Command |
|---|---|
| `listWorkItems(iteration?)` | `acli jira workitem search --jql "project=X" --json` or `acli jira sprint list-workitems --board B --sprint S --json` |
| `getWorkItem(id)` | `acli jira workitem view KEY-123 --json --fields *all` |
| `createWorkItem(data)` | `acli jira workitem create --summary --project --type [--parent] [--label] --json` |
| `updateWorkItem(id, data)` | `acli jira workitem edit --key --json` + `workitem transition --key --status` + `workitem assign --key --assignee` |
| `deleteWorkItem(id)` | `acli jira workitem delete --key --yes --json` |
| `getChildren(id)` | `acli jira workitem search --jql "parent=KEY-123" --json` |
| `getDependents(id)` | `acli jira workitem search --jql "issue in linkedIssues(KEY-123,'is blocked by')" --json` |
| `addComment(id, comment)` | `acli jira workitem comment create --key --body --json` |
| `getStatuses()` | Derived from project workflow transitions |
| `getWorkItemTypes()` | Derived from project configuration |
| `getAssignees()` | Extracted from project members or recent assignees |
| `getIterations()` | Sprint list from board |
| `getCurrentIteration()` | Active sprint from board |
| `setCurrentIteration()` | Store in `.tic/config.yml` (sprints are board-managed in Jira) |
| `getItemUrl(id)` | `{site}/browse/{key}` |
| `openItem(id)` | `acli jira workitem view --key --web` |

## Implementation Notes

### File structure
```
src/backends/jira/
├── index.ts       # JiraBackend class extending BaseBackend
├── acli.ts        # acli CLI wrapper (exec + JSON parsing)
├── mappers.ts     # Jira issue → WorkItem conversion
└── config.ts      # Jira-specific config reading from .tic/config.yml
```

### Key differences from other backends
- **No git remote detection** — Jira site/project configured explicitly in `.tic/config.yml`
- **Config extension needed** — Add optional `jira` section to Config interface
- **Update is multi-command** — Editing fields, transitioning status, and assigning are separate acli commands
- **Dependencies via issue links** — Uses Jira's link types (e.g. "Blocks"/"is blocked by") rather than a dedicated dependency field
- **Sprint iteration** — Requires `boardId` config; `setCurrentIteration` stores selection locally since sprints are managed in Jira

### Factory changes
- Add `'jira'` to `VALID_BACKENDS`
- No `detectBackend()` change needed (no remote pattern) — detection returns `'local'` fallback, user overrides via config
- Add `case 'jira'` in `createBackend()` switch

### Auth verification
Constructor calls `acli jira auth status` to verify authentication, matching the pattern used by `gh auth status`, `glab auth status`, and `az account show` in other backends.
